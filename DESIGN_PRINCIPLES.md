# Dev-Bot 设计原则和架构

## 核心哲学

### 1. 极简核心
```
while :; do
    cat PROMPT.md | iflow
done
```

**核心循环极其简单**：
- 读取提示词
- 调用 iflow
- 重复

### 2. AI 自主决策
- **iflow 必须自己决策**，不等待用户回答
- iflow 知道什么时候继续、什么时候停止、什么时候复盘
- 用户只是观察者，不参与决策

### 3. 持续优化
- iflow 可以主动优化提示词
- iflow 可以主动修改代码
- iflow 可以主动请求重启

## 架构设计

### 模块职责

#### 1. guardian.py（守护进程）
**职责**：
- 进程监控和健康检查
- 自动重启崩溃的进程
- AI 智能恢复（分析错误并自动修复）
- 资源监控（CPU/内存/磁盘）

**关键特性**：
- 检测进程崩溃
- 收集错误信息
- AI 分析错误并修复
- 重启进程

#### 2. ai_runner.py（AI 循环运行器）
**职责**：
- 运行核心循环 `while :; cat PROMPT.md | iflow; done`
- 提示词管理（动态更新）
- 代码热重载（检测代码变化）
- AI 交互日志记录

**关键特性**：
- 简单循环，不等待用户
- 检测 iflow 的指令（UPDATE_PROMPT, RESTART）
- 记录所有 AI 交互
- 自动重启以加载新代码

#### 3. iflow.py（iflow 调用器）
**职责**：
- 调用 iflow 命令
- 错误处理和分类
- 热重载指令检测

#### 4. interaction_logger.py（交互日志记录器）
**职责**：
- 记录所有 AI 交互
- 统计和分析
- 方便审计

#### 5. tui.py（文本用户界面）
**职责**：
- 实时日志显示
- 用户交互界面
- 控制面板

## 工作流程

### 核心循环
```
启动
  ↓
读取 PROMPT.md
  ↓
调用 iflow
  ↓
iflow 工作...
  ↓
[可选] iflow 优化提示词 → UPDATE_PROMPT 指令
  ↓
[可选] iflow 修改代码 → RESTART 指令
  ↓
[可选] iflow 完成 → STOP 指令
  ↓
继续循环（如果没有停止）
```

### 智能恢复流程
```
进程崩溃
  ↓
guardian 检测到崩溃
  ↓
收集错误信息
  ↓
AI 分析错误
  ↓
AI 尝试修复
  ↓
修复成功？
  ├─ 是 → 重启进程
  │
  └─ 否 → 增加重启计数
           ↓
        超过最大重启次数？
           ├─ 是 → 停止
           │
           └─ 否 → 重启进程
```

## 关键指令

### iflow 输出的指令

#### UPDATE_PROMPT
```
UPDATE_PROMPT: 改进后的提示词内容
```
触发：提示词需要优化时

#### RESTART
```
RESTART
```
触发：代码已修改，需要重启加载新代码

#### STOP
```
STOP
```
触发：任务完成，停止循环

## 设计原则

### 1. 简单优先
- 核心循环保持简单
- 避免过度设计
- 让 iflow 自己决策

### 2. AI 自主
- iflow 完全自主决策
- 不等待用户输入
- 自动优化和改进

### 3. 可观察性
- 完整的日志记录
- 统计和分析
- 方便审计

### 4. 可恢复性
- 自动检测崩溃
- AI 智能修复
- 自动重启

### 5. 可扩展性
- 模块化设计
- 清晰的职责划分
- 易于添加新功能

## 技术栈

- Python 3.9+
- asyncio（异步编程）
- iflow（AI 命令行工具）
- textual（TUI 框架）
- psutil（进程监控）
- json（配置和日志）

## 配置

### 循环配置
- 循环间隔：1 秒
- 超时时间：600 秒（10 分钟）

### 守护配置
- 最大重启次数：10 次
- 重启延迟：5 秒
- 检查间隔：30 秒

### 资源监控
- 内存阈值：80%
- CPU 阈值：90%
- 磁盘阈值：90%
