# Dev-Bot 架构 V2 - iflow 驱动系统

## 核心哲学

**Dev-Bot 的核心哲学是监督 iflow 工作，通过循环迭代，调用 iflow 完成自主迭代、自主修复、自主开发、自主进化。**

## 架构概览

```
Dev-Bot (轻量级驱动器和长期记忆)
│
├── 长期记忆系统
│   ├── 项目历史记录
│   ├── 提示词优化历史
│   ├── 开发经验总结
│   └── 学习成果保存 (.dev-bot-memory/context.json)
│
├── 驱动循环（两阶段 AI 循环）
│   ├── 阶段 1: AI 决策 (iflow --plan)
│   │   ├── 读取长期记忆
│   │   ├── 读取 spec 文件
│   │   ├── 分析当前项目状态
│   │   └── 决定下一步做什么
│   │
│   └── 阶段 2: AI 执行 (iflow -y)
│       ├── 根据决策执行任务
│       ├── 编写代码
│       ├── 修复错误
│       └── 完成功能
│
├── 监督界面 (TUI)
│   └── 显示 iflow 工作状态
│
└── 兜底机制 (AI 终端读取系统)
    ├── 检测 iflow 无限错误迭代
    ├── 自主修复（开发自身的时候）
    └── 自主重启 AI 循环
```

## 核心职责

### Dev-Bot 的职责
1. **提供长期记忆** - 保存 iflow 的历史、上下文、学习成果
2. **提供驱动循环** - 持续调用 iflow，让它不断迭代
3. **提供监督界面** - TUI 显示 iflow 工作状态和进度
4. **提供兜底机制** - AI 终端读取系统在最底层保障程序运行
5. **Spec 驱动** - 通过 spec 引导 iflow 进行有目的的开发

### iflow 的职责
1. **自主决策** - 分析项目，决定下一步做什么
2. **自主执行** - 根据决策执行任务
3. **自主修复** - 发现问题，自动修复
4. **自主开发** - 实现功能，添加代码
5. **自主进化** - 优化架构，提升能力

## 两阶段 AI 循环

### 阶段 1: AI 决策
```bash
iflow --plan "分析当前项目状态，决定下一步要做什么"
```

- 使用 `--plan` 参数
- 只允许 iflow 做决策和规划
- 不执行任何修改操作
- 输出决策结果和实现策略

### 阶段 2: AI 执行
```bash
iflow -y "根据决策执行任务，编写代码，修复错误"
```

- 使用 `-y` 参数（yolo 模式）
- 授权 iflow 执行操作无需用户授权
- iflow 可以直接修改代码
- iflow 可以直接执行命令

## iflow 参数说明

| 参数 | 说明 | 互斥 | 使用场景 |
|------|------|------|----------|
| `--plan` | 只做计划，不执行 | 与 `-y` 互斥 | AI 决策阶段 |
| `-y` | yolo 模式，无需授权 | 与 `--plan` 互斥 | AI 执行阶段 |
| `--thinking` | 思考模式，更全面 | 无 | 复杂任务分析 |

```
启动 Dev-Bot

iflow 开始自主工作
```

## 自主进化的含义

### 1. 自我迭代开发（针对自身项目）
- Dev-Bot 自己修改自己的工程代码
- 自己添加新功能
- 自己修复 bug
- 自己优化架构
- 自己提升能力

### 2. 优化提示词（针对其他项目）
- 为其他项目生成和优化提示词
- 根据项目特点调整 iflow 的工作方式
- 迭代改进开发流程
- 提升其他项目的开发效率

## 兜底机制

### 三大作用

1. **发现 iflow 无限错误迭代**
   - 监控 iflow 的工作状态
   - 检测到 iflow 陷入错误循环
   - 防止无限重复相同的错误操作
   - 及时中断并恢复

2. **自主修复**（开发自身的时候）
   - 当 Dev-Bot 运行时发生错误
   - AI 终端读取系统检测到错误
   - iflow 自主修复自己的代码
   - 恢复正常运行

3. **自主重启 AI 循环**
   - 当 AI 循环意外终止
   - AI 终端读取系统检测到终止
   - 自动重启 AI 循环
   - 确保持续工作

### AI 循环流程

```python
async def ai_loop():
    while not shutdown_requested:
        # 加载长期记忆
        memory = load_memory()
        
        # 读取 spec
        spec_content = read_spec()
        
        # ========== 阶段 1: AI 决策 ==========
        decision_prompt = f"""
当前上下文：{memory['context']}
Spec 内容：{spec_content}
分析当前项目状态，决定下一步要做什么。
"""
        
        decision = await call_iflow(
            prompt=decision_prompt,
            args=["--plan"]  # 只做计划
        )
        
        # ========== 阶段 2: AI 执行 ==========
        execution_prompt = f"""
决策结果：{decision}
根据决策执行任务，编写代码，修复错误。
"""
        
        result = await call_iflow(
            prompt=execution_prompt,
            args=["-y"]  # yolo 模式
        )
        
        # 保存结果到长期记忆
        memory['history'].append({
            "session": session_num,
            "phase": "decision",
            "content": decision
        })
        memory['history'].append({
            "session": session_num,
            "phase": "execution",
            "content": result
        })
        
        save_memory(memory)
```

## 核心优势

1. **清晰的分离** - 决策和执行分开，职责明确
2. **灵活的决策** - iflow 可以根据情况做出不同的决策
3. **高效的执行** - 有了明确的决策，执行更加专注
4. **持续迭代** - 每轮循环都基于前一轮的结果
5. **完全自主** - 无需人工干预
6. **Spec 驱动** - 所有工作都基于 spec
7. **长期记忆** - 持续积累知识和经验

## 总结

Dev-Bot 现在是一个轻量级的 iflow 驱动器和长期记忆系统：

- **Dev-Bot 不做决策** - 所有决策由 iflow 做出
- **Dev-Bot 不执行代码** - 所有代码修改由 iflow 完成
- **Dev-Bot 只驱动** - 提供循环和上下文
- **iflow 是主角** - Dev-Bot 只是 iflow 的助手

这样的架构更清晰、更简单，充分发挥 iflow 作为 AI Agent 的能力！
## 2026-03-08 架构优化：AI 循环执行流程改进

### 优化背景
原有 AI 循环存在执行阶段过早切换的问题，导致：
1. 上下文频繁清空，工作中断
2. 每次执行只完成小部分工作，效率低下
3. 工作未完成就进入下一轮，影响连续性

### 优化后的架构

#### 1. 两阶段循环设计

**执行阶段（Execution Phase）**
```
输入：项目状态 + 记忆摘要
过程：
  1. AI 自主分析项目状态
  2. 识别需要完成的工作
  3. 制定执行计划
  4. 在一次调用中完成所有相关工作
  5. 评估完成度（0-100%）
输出：执行结果 + 完成度 + 结论（[工作完成]/[继续执行]）
```

**复盘阶段（Review Phase）**
```
输入：执行结果 + 记忆摘要
过程：
  1. 评估工作完成度
  2. 分析工作质量和效率
  3. 识别问题和改进空间
  4. 决定下一步行动
决策：
  - 完成度 >= 80% 且无重大问题 → [工作完成]
  - 完成度 < 80% 或存在重大问题 → [继续执行]
```

#### 2. 循环控制逻辑

```
while ai_loop_running:
    # 执行阶段
    result = await ai.execute()
    
    # 复盘阶段
    review = await ai.review(result)
    
    # 决策
    if review.decision == "完成":
        execution_results = []  # 清空上下文
    else:
        # 保留上下文，继续执行
        pass
```

#### 3. 提示词架构

**基础提示词（Base Prompt）**
```
项目信息
- 项目路径
- 技术栈
- 代码风格

工作原则
- 先分析，后行动
- 小步快跑，频繁验证
- 遇到错误立即停止
- 修改代码后必须测试
- 代码审查

输出格式
- [分析]
- [决策]
- [执行]
- [验证]
- [结论]

停止条件
- 所有功能已实现且测试通过
- 连续3次遇到相同错误无法解决
- 需要用户决策或输入
- 接收到停止信号

安全规则
- 不要删除重要文件
- 不要提交未经测试的代码
- 不要修改配置文件（除非明确需要）
- 不要运行危险的系统命令
- 修改代码前先备份
```

**执行阶段提示词（Execution Prompt）**
```
工作任务
分析项目状态，识别需要完成的工作，制定执行计划，
并尽可能在本次调用中完成所有工作。

执行要求
1. 自主决策：根据项目实际情况，自主决定要完成什么工作
2. 一次性完成：在本次调用中尽可能完成所有相关工作，不要分步请求
3. 详细记录：记录工作目标、执行过程、操作结果
4. 评估完成度：0-100% 评估工作完成程度
5. 明确结论：完成后输出 [工作完成]，否则输出 [继续执行]

输出格式
工作目标：...
执行计划：...
执行过程：
- 步骤1：... 结果：...
- 步骤2：... 结果：...
执行结果：...
完成度：X%
结论：[工作完成] 或 [继续执行]
```

**复盘阶段提示词（Review Prompt）**
```
复盘任务
评估执行阶段的工作，判断工作是否完成，并决定下一步行动。

评估维度
1. 工作完成度：0-100%，目标是否达成
2. 工作质量：代码规范、逻辑正确性
3. 执行效率：是否高效完成
4. 问题发现：错误、遗漏、不足

决策标准
- 工作完成（完成度 >= 80% 且无重大问题）：输出 [工作完成]
- 继续执行（完成度 < 80% 或存在重大问题）：输出 [继续执行]

输出格式
工作完成度：X%
质量评估：...
问题分析：...
改进建议：...
决策结论：[工作完成] 或 [继续执行]
后续计划：...
```

#### 6. 决策机制

**执行阶段决策**
- AI 自主决定完成什么工作
- AI 自主决定是否完成
- 输出明确的结论标记

**复盘阶段决策**
- 评估完成度（>= 80% 为完成）
- 检查重大问题
- 输出决策结论

**上下文管理决策**
- 工作完成：清空执行结果
- 工作未完成：保留执行结果

### 关键设计原则

1. **一次性完成**：AI 在一次调用中完成整个工作
2. **评估驱动**：通过复盘评估决定是否继续
3. **上下文连续**：未完成的工作保留上下文
4. **明确决策**：使用明确的标记表示决策
5. **简化流程**：减少不必要的判断和切换

### 性能优化

1. **减少切换**：从频繁切换改为一次调用完成
2. **减少重复**：保留上下文，避免重复分析
3. **提高效率**：简洁的提示词，降低 token 消耗
4. **加速收敛**：明确的决策标准，快速达到目标
