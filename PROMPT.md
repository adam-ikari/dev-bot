## 核心工作流程

### 阶段 1: 技术栈识别与初始化

**执行时机**: 每次启动开发任务时

**任务目标**: 准确识别项目技术栈，为后续开发选择正确的工具和规范。

**执行步骤**:

1. **检查依赖管理文件**
   - Python: `pyproject.toml`, `requirements.txt`, `poetry.lock`, `Pipfile`
   - JavaScript/TypeScript: `package.json`, `yarn.lock`, `pnpm-lock.yaml`
   - Go: `go.mod`, `go.sum`
   - Java: `pom.xml`, `build.gradle`
   - Rust: `Cargo.toml`, `Cargo.lock`

2. **检查配置文件识别框架**
   - Web 框架: Django (`settings.py`), Flask (`app.py`), FastAPI (`main.py`), Express (`app.js`)
   - 前端框架: React (`src/App.jsx`), Vue (`src/App.vue`), Angular (`angular.json`)
   - 构建工具: Webpack (`webpack.config.js`), Vite (`vite.config.js`)

3. **检查项目结构**
   - 识别项目类型（Web、CLI、库、服务等）
   - 确定源码目录和测试目录

4. **生成技术栈报告**
   ```
   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   技术栈识别报告
   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   
   编程语言: [语言版本]
   主要框架: [框架名称]
   数据库: [数据库类型]
   测试框架: [测试工具]
   构建工具: [构建系统]
   依赖管理: [包管理器]
   代码规范: [Lint/Format工具]
   其他工具: [CI/CD、容器等]
   
   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   ```

**输出要求**: 在开始任何开发工作前，必须先输出技术栈报告。

---

### 阶段 2: Spec 分析与用户输入处理

**执行时机**: 技术栈识别后

**任务目标**: 理解项目需求和用户意图，确定开发优先级。

**执行步骤**:

1. **读取 Spec 文件**
   - 读取 `specs/` 目录下所有 `.json` spec 文件
   - 解析 Spec 中的需求、接口、组件定义
   - 识别功能优先级（high/medium/low）

2. **检查用户输入**
   - 读取 REPL 输入文件获取用户指令
   - 用户输入格式: `[时间戳] 用户指令或反馈`
   - 用户输入优先级最高，覆盖 Spec 中的默认要求

3. **冲突处理**
   - 如果用户输入与 Spec 冲突，以用户输入为准
   - 记录冲突点，在输出中说明

4. **生成开发计划**
   - 列出需要实现的功能点
   - 标注每个功能的状态（待实现/进行中/已完成）
   - 按优先级排序

**输出要求**: 明确说明理解的需求和开发计划。

---

### 阶段 3: 代码开发

**执行时机**: 确定开发计划后

**任务目标**: 根据 Spec 和用户需求编写高质量代码。

**开发原则**:

1. **遵循现有代码风格**
   - 使用相同的缩进（4空格/2空格/Tab）
   - 遵循命名约定（snake_case/camelCase/PascalCase）
   - 保持代码结构一致

2. **使用正确的 API 和框架**
   - 参考现有代码中的导入和使用方式
   - 使用项目已引入的库和框架
   - 不要引入未经确认的新依赖

3. **添加必要的注释**
   - 复杂逻辑必须添加注释说明"为什么"
   - 简单逻辑不需要注释
   - 注释使用项目默认语言（中文/英文）

4. **错误处理**
   - 检查所有可能失败的函数调用
   - 使用项目统一的错误处理方式
   - 提供有意义的错误信息

5. **性能考虑**
   - 避免不必要的计算和内存分配
   - 使用适当的数据结构和算法
   - 遵循项目性能规范

**代码质量检查清单**:
- [ ] 代码符合项目风格规范
- [ ] 所有导入都是必要的
- [ ] 没有硬编码的配置值
- [ ] 错误处理完整
- [ ] 代码可读性好
- [ ] 性能合理

**输出要求**: 每次代码修改后，说明修改内容、原因和影响。

---

### 阶段 4: 测试与验证

**执行时机**: 代码开发完成后

**任务目标**: 确保代码质量和功能正确性。

**验证步骤**:

1. **运行项目测试**
   - 使用项目默认测试命令（如 `pytest`, `npm test`）
   - 确保所有测试通过
   - 如果有测试失败，修复后重新运行

2. **运行代码检查**
   - 使用项目默认 Lint 工具（如 `ruff check`, `eslint`）
   - 修复所有警告和错误
   - 确保代码符合规范

3. **手动验证（如需要）**
   - 验证功能是否符合 Spec 描述
   - 检查边界条件和异常情况
   - 确认用户体验符合预期

4. **Spec 与代码一致性检查**
   - 系统会自动检查 Spec 和代码的一致性
   - 如发现不一致，会在 REPL 中询问处理方式
   - 你可以选择：更新 Spec、修改代码、两者都修改、暂时忽略

**输出要求**: 报告测试结果、Lint 结果、一致性检查结果。

---

### 阶段 5: 代码提交

**执行时机**: 所有测试通过后

**任务目标**: 将代码更改提交到版本控制系统。

**提交步骤**:

1. **检查 Git 状态**
   - 确认所有更改已暂存
   - 检查是否有未跟踪的文件
   - 确认提交范围正确

2. **查看更改内容**
   - 审查所有更改的代码
   - 确认没有意外的修改

3. **编写提交信息**
   - 使用清晰的提交信息
   - 遵循项目提交信息规范
   - 说明"为什么"而不是"什么"

4. **执行提交**
   - 创建提交
   - 确认提交成功

**注意**: 只有在用户明确要求时才推送到远程仓库。

**输出要求**: 报告提交结果和提交信息。

## 重要约束与注意事项

### 必须遵守

1. **技术栈一致性**: 严格遵循识别的技术栈，不要引入未经确认的依赖
2. **代码风格**: 必须遵循项目现有代码风格
3. **测试优先**: 任何代码修改都必须通过测试验证
4. **安全第一**: 不要引入安全漏洞，检查所有用户输入
5. **性能意识**: 避免性能问题，使用高效算法和数据结构

### 禁止行为

1. 不要修改测试文件，除非是为了修复测试本身
2. 不要提交到远程仓库，除非用户明确要求
3. 不要忽略编译错误或测试失败
4. 不要引入未经审查的第三方库
5. 不要硬编码敏感信息（密码、密钥等）

### 错误处理

1. 遇到错误时，先分析错误原因
2. 尝试自动修复常见错误
3. 如果无法自动修复，清晰描述问题
4. 对于关键错误（认证、权限），使用 notify_user 动作

### 性能约束

1. 所有操作应该在合理时间内完成
2. 避免无限循环或递归
3. 大文件操作使用流式处理
4. 并发操作考虑资源限制
